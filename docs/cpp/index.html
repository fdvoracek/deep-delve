{ } Deep Delve — C++-First

Build the whole game in C++ from Quest 1, with thin Blueprints that only assign data.

The working pattern

Every system is a C++ class. You then create a Blueprint subclass (BP_… / WBP_…) whose only job is to assign values to the class's UPROPERTYs — meshes, sounds, input assets, and UPrimaryDataAsset references — and to drop the actor into the level. UMG uses C++ UUserWidget bases with meta=(BindWidget) controls and WBP_ layouts. Data lives in DA_ assets.

The full quest log spells out, per quest: the C++ classes & methods created/updated, the matching Blueprints, and a UPROPERTY → value table for what you assign in each Blueprint.

🧱 Architecture at a glance self-contained

The chain is dependency-audited — every class, method, asset and UPROPERTY is introduced before it's used, so you never need to build anything outside the quests. Key decisions:

  • The economy subsystem owns the state — UMineEconomySubsystem holds Coins, Depth, tech NodeLevels, EquippedPickaxe, PickaxeDamageBonus, CurrentRock/CurrentRockActor, and the crew/collector multipliers. No Blueprint needed.
  • Player power = EquippedPickaxe->Damage + PickaxeDamageBonus, read via the controller's GetDamage().
  • One swing path — PerformSwing() (manual + auto-swing) rolls crits and calls the shared ShowDamageNumber() helper; dwarves call the same door. No UI code lives in the subsystem.
  • Everything mineable goes through the IMineable interface — clicks, dwarves, and auto-swing all call TakeDamage.
  • Descend actually works — SetCurrentRock() swaps the live rock actor and nudges the camera; AOreVein always reads the subsystem's CurrentRock.
  • Data-driven — rocks, pickaxes, miners and tech nodes are UPrimaryDataAssets; effects are instanced UTechEffects. The SaveGame uses TSoftObjectPtr and records the hired crew so it rebuilds the world exactly.
  • Build.cs modules: EnhancedInput (Q2), UMG/Slate/SlateCore (Q4), Niagara (Q16).
